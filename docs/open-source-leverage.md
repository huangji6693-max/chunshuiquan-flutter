# 春水圈可优先借力的开源方向

目标：避免从 0 到 1 重造轮子，优先利用成熟开源项目做“从 1 到 2”的改造。

## 已经在用 / 可继续借力

### 1. Flutter Chat UI
- 仓库：`flyerhq/flutter_chat_ui`
- 用途：聊天消息列表、输入框、气泡、空态等 UI 层能力
- 当前状态：仓库已接入 `flutter_chat_ui`
- 建议：继续把它当作聊天前端基础设施，而不是自研消息 UI

## 建议重点调研

### 2. 开源 Flutter Dating App 结构参考
- 候选：`helloharendra/Complete-Dating-App`
- 可借鉴点：
  - 滑卡产品流程拆分
  - 配对后的聊天入口设计
  - 资料页、推荐页、状态页的组织方式
- 使用原则：借鉴模块拆分和交互流，不直接照搬业务实现

### 3. Spring Boot + Kafka + Redis 微服务参考骨架
- 关键词：`spring-boot-kafka`, `spring boot redis kafka microservices`
- 可借鉴点：
  - 事件驱动消息链路
  - 聊天/通知异步化
  - Redis 缓存层设计
  - 微服务间事件解耦
- 使用原则：优先挑 star 高、结构清晰、最近仍活跃的项目，只吸收基础设施层方案

## 春水圈建议的借力策略

1. **聊天 UI 不自研**
   - 继续基于 `flutter_chat_ui` 做定制

2. **滑卡/社交交互参考成熟 dating app 开源项目**
   - 借它们的页面结构、状态流和交互细节

3. **后端基础设施优先吸收成熟 Spring Cloud 方案**
   - 注册中心 / 配置中心：Nacos
   - 缓存：Redis
   - 消息队列：Kafka 优先
   - 定时任务：XXL-JOB
   - 对象存储：AWS S3
   - 数据库：TiDB

4. **禁止重复造轮子**
   - 通用聊天组件、上传流程、消息收敛、任务调度、配置管理、监控链路，优先找成熟开源方案

## 执行原则

- 先找开源参考，再决定是否自己写
- 如果成熟项目能覆盖 70% 以上，就做集成和裁剪，不做重写
- 只在业务强定制部分做自研
